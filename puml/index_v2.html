🎯 The Problem We're Solving

Data teams spend 40-60% of their time fighting fires: investigating pipeline failures, tracking down data quality issues, and manually writing validation rules. This reactive approach leads to:

  • Delayed detection: Issues are found downstream, after business impact
  • Manual diagnosis: Engineers spend hours digging through logs and metrics
  • Repetitive work: Writing similar validation rules over and over
  • Inconsistent quality: Each team has different standards and tools

💡 Our Solution: The 3D Framework

DRE 3D is a self-driving data reliability engine that combines AI agents with Databricks platform to automate the entire data quality lifecycle:

🤖 Self-Service Rule Portal: Capture data quality rules via chatbot in plain English - no coding required!
Example: "Alert me if customer_id is null in orders table" → System generates SQL + validation logic
D1 - Detect: Runs user-defined rules on schedule to monitor pipelines and detect anomalies in real-time
D2 - Diagnose: When anomalies are found, AI investigates root causes by correlating logs, metrics, and deployments
D3 - Deflect: Generates fixes, creates tickets, and can auto-remediate with approval

Key Innovation: The self-service portal democratizes data quality. Business analysts and data engineers can define rules through a conversational chatbot interface without writing code. The system validates rules through 10 layers (security, governance, duplicate detection, SQL validation) before deploying them to production where D1 Detection Agent executes them continuously.

The Complete Flow: User defines rule via chatbot → LLM translates to SQL → Multi-stage validation → Approval → Deployed to D1 → D1 runs rule on schedule → Detects anomaly → D2 diagnoses → D3 fixes.

👤 User Journeys: What Does It Look Like?

Two main ways users interact with the system: Interactive (self-service portal with chatbot to create rules and ask questions) and Async (scheduled D1/D2/D3 jobs running in background).

Journey 1: Self-Service Rule Creation via Chatbot

The story: Sarah (data analyst, no SQL knowledge) opens the self-service portal and types in the chatbot:

"Check if customer_id is never null in orders table"

The LLM translates this to SQL, validates through 10 stages (security, PII check, duplicate detection), generates the final DQ rule, and routes for approval. Once approved, D1 Detection Agent uses this rule to continuously monitor the orders table.

Key benefit: No coding required. Rules are captured conversationally and automatically deployed to D1.

View Source

Journey 2: Asking "Why Did My Pipeline Fail?"

The story: John (data engineer) sees a failed pipeline and asks the chatbot:

"Why did the customer_orders pipeline fail at 3am?"

D2 Diagnosis Agent retrieves logs from the last 7 days, correlates with recent deployments in last 24 hours, checks data drift, consults incident knowledge base, and generates a root cause report with fix recommendations.

View Source

Journey 3: Automatic Detection & Deflection

The story: Every hour, D1 Detection Agent runs all user-defined quality rules. When it finds an anomaly...

D1 detects null customer_id (Sarah's rule from Journey 1) → D2 automatically investigates root cause → D3 generates a fix (e.g., "add NOT NULL constraint"), creates a Jira ticket with fix code, and waits for approval before applying the fix.

Key benefit: Full automation loop from rule creation (Journey 1) → detection (D1) → diagnosis (D2) → deflection (D3).

🛡️ Safety & Governance: How We Prevent Disasters

The most important part: We don't just generate SQL from natural language and run it. Every rule goes through 10 validation stages to prevent security issues, PII leaks, and duplicate work.

10-Stage Validation Pipeline

What it shows: The journey of a user's request from raw query to approved SQL

Key stages:

  • Query Optimization (LLM): Clarifies ambiguous requests
  • Security Layer (Rule-based): Blocks DROP, DELETE, ALTER commands
  • Governance Layer (RAG): Checks for PII data access
  • Duplicate Detection (Vector Search): "This rule already exists as Rule #143"
  • SQL Validation (Rule-based): Syntax check, table existence
  • Approval Gateway: Routes to right approver based on impact
